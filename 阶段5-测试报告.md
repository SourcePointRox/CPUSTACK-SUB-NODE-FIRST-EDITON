## 3. 交付模块清单

### 3.1 新增模块

#### 3.1.1 算法优化参数构建器

- **文件**：[worker/backends/params.py](file:///f:/CPUSTACK-设计/cpustack/backend/cpustack/worker/backends/params.py)
- **职责**：从 `Model.backend_parameters` JSON 构建推理后端命令行参数
- **核心 API**：
  - `build_common_args(params, instance) -> list[str]`：构建通用参数（ctx/threads/batch/flash_attn/mlock/cont_batching/cache_reuse/draft_model/extra_args）
  - `parse_backend_parameters(raw) -> dict`：兼容 JSON 字符串/dict/None
- **设计要点**：
  - 全部参数可选，缺省保持阶段4 行为（向后兼容）
  - 草稿模型路径不存在则跳过投机解码（避免启动失败）
  - `extra_args` 仅透传不做合法性校验（运维责任）

#### 3.1.2 放置打分器

- **文件**：[server/scheduler/placement.py](file:///f:/CPUSTACK-设计/cpustack/backend/cpustack/server/scheduler/placement.py)
- **职责**：三维度放置打分 + RPC Master 选择 + 流水线节点选择
- **核心 API**：
  - `score_placement(candidates, strategy, model, backend_parameters) -> (Worker, WorkerStatus) | None`
  - `select_pipeline_nodes(candidates, node_count, model) -> list[(Worker, WorkerStatus)]`
  - `select_rpc_master(candidates) -> (Worker, WorkerStatus) | None`
- **打分维度（优先级高到低）**：
  1. 指令集能力（AMX=100 > AVX-512=50 > AVX-VNNI=30 > AVX2=10）
  2. 网络带宽（仅流水线并行，对数缩放：100Mbps≈46, 1Gbps≈69, 10Gbps≈92）
  3. 内存分配策略（SPREAD 选最空 / BINPACK 选最满）
- **策略覆盖**：`backend_parameters.placement_strategy` 可覆盖默认 `spread`

#### 3.1.3 性能基准测试框架

- **文件**：[benchmark.py](file:///f:/CPUSTACK-设计/cpustack/backend/cpustack/benchmark.py)
- **职责**：吞吐/延迟/资源利用率基准测试
- **核心数据结构**：
  - `RequestResult`：单次请求结果（success/latency_ms/ttft_ms/tokens）
  - `BenchmarkReport`：完整报告（throughput_tok_per_s/request_qps/P50/P95/P99/avg_ttft/p95_ttft/error_breakdown/resource_snapshot）
- **CLI 用法**：

```bash
# 非流式基准
python -m cpustack.benchmark --endpoint http://localhost:80/v1 \
    --model llama-3.2-3b --api-key sk-xxx --concurrency 4 --duration 60

# 流式 TTFT 测试
python -m cpustack.benchmark --endpoint http://localhost:80/v1 \
    --model llama-3.2-3b --api-key sk-xxx --stream --duration 30

# 输出 JSON 报告
python -m cpustack.benchmark ... --report report.json
```

## 4. 关键设计验证

### 4.1 向后兼容性

阶段5 所有优化参数均为**可选**，未配置 `backend_parameters` 时保持阶段4 行为：

| 场景 | 阶段4 行为 | 阶段5 行为（未配置参数） | 验证结果 |
|------|-----------|----------------------|---------|
| 单机后端启动 | `--ctx-size 4096 --threads 4` | 同左（默认值一致） | ✅ 测试1 |
| RPC 调度 | 按内存降序选 Master | 同左（`select_rpc_master` 单候选直接返回） | ✅ 测试8 |
| 流水线调度 | 按 CPU 核心降序 | 同左（指令集相同时 CPU 核心打分） | ✅ 测试7 |
| 放置策略 | 无策略 | 默认 `spread`（选最空节点） | ✅ 测试5 |

### 4.2 安全性设计

| 风险 | 设计 | 验证结果 |
|------|------|---------|
| 草稿模型路径不存在导致启动失败 | 跳过投机解码并记录警告 | ✅ 测试3 |
| `backend_parameters` JSON 格式错误 | `parse_backend_parameters` 返回空 dict | ✅ 测试4 |
| `placement_strategy` 非法值 | 回退到默认策略 | ✅ 测试5 |
| 空候选节点列表 | `score_placement` 返回 None，实例保持 PENDING | ✅ 测试5 |
| 流水线节点带宽不足 | 记录警告日志（不阻断调度） | ✅ 测试7 |

### 4.3 调度策略三维度打分

`score_placement` 排序优先级验证：

```
场景：3 个候选节点
  A: AVX2,    内存已分配 0MB    （指令集=10, 网络=0, mem=-0）
  B: AVX-512, 内存已分配 4096MB （指令集=50, 网络=0, mem=-4096）
  C: AMX,     内存已分配 8192MB （指令集=100, 网络=0, mem=-8192）

预期选择：C（AMX 最高优先级，即使内存已分配最多）
实测选择：C ✅
```

→ 验证结论：指令集维度优先于内存策略，符合规划文档 §2.3「AVX-512 较 AVX2 提升 23%+，AMX 革命性提升」的设计意图。

```bash

    --model llama-3.2-3b --api-key sk-xxx \
    --concurrency 4 --duration 60 --report single.json

# RPC 分布式基准（qwen2.5-7b，多节点内存池化）

    --model qwen2.5-7b --api-key sk-xxx \
    --concurrency 4 --duration 60 --report rpc.json

    --model llama-3.2-3b --api-key sk-xxx \
```

- `throughput_tok_per_s`：生成 token 吞吐

- `avg_ttft_ms` / `p95_ttft_ms`：首 token 延迟（流式）
- `resource_snapshot`：CPU/内存采样（需安装 psutil）
