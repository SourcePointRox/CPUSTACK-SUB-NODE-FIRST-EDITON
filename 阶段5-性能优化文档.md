## 1. 优化维度总览

CPUSTACK 阶段5 性能优化覆盖三大维度，全部通过 `Model.backend_parameters` JSON 字段配置，**全部可选，缺省保持阶段4 行为**：

```
┌─────────────────────────────────────────────────────────┐
│           backend_parameters JSON 配置层                  │
├─────────────┬─────────────┬─────────────────────────────┤
│  算法优化    │  调度优化    │       运维扩展              │
│  (§2)       │  (§3)       │       (§4)                 │
├─────────────┼─────────────┼─────────────────────────────┤
│ cont_batching│ placement_ │ extra_args                 │
│ flash_attn   │ strategy   │ (透传任意 llama-server 参数)│
│ mlock        │ (spread/   │                            │
│ cache_reuse  │  binpack)  │                            │
│ draft_model  │            │                            │
│ n_gpu_layers │            │                            │
└─────────────┴─────────────┴─────────────────────────────┘
```

## 2. 算法优化参数

### 2.1 参数清单

通过 `Model.backend_parameters` JSON 配置，由 [worker/backends/params.py](file:///f:/CPUSTACK-设计/cpustack/backend/cpustack/worker/backends/params.py) 的 `build_common_args()` 解析为命令行参数。

#### 2.1.1 基本参数

| 参数 | 类型 | 默认值 | 命令行 | 说明 |
|------|------|--------|--------|------|
| `ctx_size` | int | 4096 | `--ctx-size` | 上下文窗口大小 |
| `threads` | int | `allocated_cpu_cores` 或 4 | `--threads` | 推理线程数 |
| `n_batch` | int | — | `--batch-size` | 批处理大小 |
| `n_ubatch` | int | — | `--ubatch-size` | 微批处理大小 |

#### 2.1.2 性能优化开关

| 参数 | 类型 | 默认 | 命令行 | 预期收益 | 适用场景 |
|------|------|------|--------|---------|---------|
| `flash_attn` | bool | false | `--flash-attn` | 长序列内存降 30%+，注意力计算加速 | 长上下文（>4K）场景 |
| `mlock` | bool | false | `--mlock` | 防止 swap 导致性能抖动 | 内存充足的生产环境 |
| `cont_batching` | bool | false | `--cont-batching` | **吞吐 2-5×** | 高并发场景 |
| `parallel` | int | 0 | `--parallel` | 配合 `cont_batching`，指定并行序列数 | 高并发场景 |
| `cache_reuse` | int | — | `--cache-reuse` | TTFT 降 40%+ | 多轮对话/系统提示词复用 |
| `n_gpu_layers` | int | 0 | `--n-gpu-layers` | 异构节点 GPU 卸载（CPU 平台默认 0） | CPU+GPU 混合节点 |

#### 2.1.3 投机解码（降延迟 2-3×）

| 参数 | 类型 | 默认 | 命令行 | 说明 |
|------|------|------|--------|------|
| `draft_model` | str | — | `--model-draft` | 草稿模型路径（**必须存在才启用**） |
| `draft_max_tokens` | int | 16 | `--draft-max` | 单次投机最大 token 数 |
| `draft_min_p` | float | — | `--draft-min-p` | 草稿接受最小概率 |

> **安全设计**：`draft_model` 路径不存在时跳过投机解码并记录警告，避免后端启动失败（见 [params.py#L114-L129](file:///f:/CPUSTACK-设计/cpustack/backend/cpustack/worker/backends/params.py)）。

#### 2.1.4 运维扩展

| 参数 | 类型 | 说明 |
|------|------|------|
| `extra_args` | `list[str]` | 透传任意 llama-server/prima-server 参数，如 `["--no-mmap", "--numa"]` |

> `extra_args` 仅追加不做合法性校验，由运维承担责任。

### 2.2 配置示例

#### 示例 1：高并发场景（连续批处理）

```json
{
  "ctx_size": 8192,
  "threads": 8,
  "cont_batching": true,
  "parallel": 16,
  "flash_attn": true,
  "mlock": true
}
```

**预期效果**：
- 吞吐提升 2-5×（连续批处理）
- 长上下文内存降低 30%+（Flash Attention）
- 无 swap 抖动（mlock）

#### 示例 2：低延迟场景（投机解码）

```json
{
  "ctx_size": 4096,
  "threads": 4,
  "draft_model": "/models/llama-3.2-1b-instruct-q4_k_m.gguf",
  "draft_max_tokens": 32,
  "draft_min_p": 0.9,
  "flash_attn": true
}
```

**预期效果**：
- 端到端延迟降低 2-3×（投机解码）
- 草稿模型用 1B 小模型，目标模型用 3B+ 大模型

#### 示例 3：多轮对话场景（前缀缓存复用）

```json
{
  "ctx_size": 16384,
  "cache_reuse": 256,
  "cont_batching": true,
  "parallel": 4,
  "flash_attn": true
}
```

**预期效果**：
- TTFT 降低 40%+（系统提示词和上下文复用 KV Cache）
- 适合客服机器人、Agent 等固定系统提示词场景

#### 示例 4：流水线并行 + 优化

```json
{
  "ctx_size": 4096,
  "threads": 4,
  "flash_attn": true,
  "total_layers": 32,
  "placement_strategy": "spread"
}
```

**说明**：`total_layers` 由 `prima_cpp` 后端读取用于层切片，`placement_strategy` 覆盖调度策略。

## 3. 资源调度优化

### 3.1 三维度放置打分

由 [server/scheduler/placement.py](file:///f:/CPUSTACK-设计/cpustack/backend/cpustack/server/scheduler/placement.py) 的 `score_placement()` 实现，**排序优先级从高到低**：

#### 维度 1：指令集能力（性能分水岭）

| 指令集 | 权重 | 说明 |
|--------|------|------|
| AMX | 100 | Intel 至强 SPR+，革命性提升（规划文档 §2.3） |
| AVX-512 | 50 | 较 AVX2 提升 23%+ |
| AVX-VNNI | 30 | 中间档 |
| AVX2 | 10 | 基准 |

**设计意图**：即使 AVX-512 节点内存已分配更多，也优先于 AVX2 节点（性能差距大于负载均衡收益）。

#### 维度 2：网络带宽（仅流水线并行）

| 带宽 | 得分 | 评估 |
|------|------|------|
| 100Mbps | 46 | 绝对不足（规划文档 §2.4） |
| 1Gbps | 69 | 勉强可用 |
| 10Gbps | 92 | 起步门槛 |
| 25Gbps+ | 100+ | 推荐 |

- **仅流水线并行（`PRIMA_CPP`）感知网络**：RPC 模式是内存池化，对网络延迟不敏感
- **低带宽警告**：节点带宽 < 10Gbps 时记录警告日志（不强制过滤，避免死锁）

#### 维度 3：内存分配策略

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `spread`（默认） | 选 `memory_allocated` 最少的节点 | 最大化可用性，副本分散 |
| `binpack` | 选 `memory_allocated` 最多的节点 | 打包腾出大内存节点 |

**策略覆盖**：`backend_parameters.placement_strategy` 可覆盖默认 `spread`：

```json
{
  "placement_strategy": "binpack"
}
```

### 3.2 RPC Master 选择

`select_rpc_master()` 综合两个维度选择 Master 节点：

1. **指令集能力**（主维度）
2. **可用内存**（次维度，同指令集时选大内存）

**与阶段4 差异**：阶段4 仅按可用内存降序选择 Master，阶段5 优先选择指令集强的节点作为 Master（Master 加载模型主体，性能影响最大）。

### 3.3 流水线节点选择

`select_pipeline_nodes()` 综合三个维度选择 N 个节点：

1. **指令集能力**（主维度）
2. **网络带宽**（次维度，流水线对延迟敏感）
3. **CPU 核心数**（末维度，核心多的节点处理更多层）

**与阶段4 差异**：阶段4 仅按 CPU 核心降序选择，阶段5 优先选择高带宽、强指令集节点组合，降低流水线气泡。

### 4.1 A/B 测试方案

使用 [benchmark.py](file:///f:/CPUSTACK-设计/cpustack/backend/cpustack/benchmark.py) 对同一模型分别测试 A 组（关闭优化）和 B 组（开启优化），对比报告字段：

| 对比项 | A 组配置 | B 组配置 | 关键指标 | 预期收益 |
|--------|---------|---------|---------|---------|
| 连续批处理 | `{}` | `{"cont_batching":true, "parallel":8}` | `throughput_tok_per_s` | 2-5× |
| Flash Attention | `{}` | `{"flash_attn":true}` | `p95_latency_ms`（长序列） | 30%+ 降 |

| 前缀缓存复用 | `{}` | `{"cache_reuse":256}` | `avg_ttft_ms` | 40%+ 降 |
| 内存锁定 | `{}` | `{"mlock":true}` | `p99_latency_ms` 波动 | 抖动消除 |

## 5. 调优建议

### 5.1 场景化推荐配置

#### 场景 A：单用户低延迟对话

```json
{
  "ctx_size": 4096,
  "threads": 4,
  "flash_attn": true,
  "draft_model": "/models/llama-3.2-1b-instruct-q4_k_m.gguf",
  "draft_max_tokens": 16
}
```

**目标**：最小化首 token 延迟和端到端延迟。

#### 场景 B：多用户高并发服务

```json
{
  "ctx_size": 8192,
  "threads": 8,
  "cont_batching": true,
  "parallel": 16,
  "flash_attn": true,
  "mlock": true,
  "cache_reuse": 256
}
```

**目标**：最大化吞吐，牺牲单请求延迟。

#### 场景 C：长上下文文档处理

```json
{
  "ctx_size": 32768,
  "threads": 8,
  "flash_attn": true,
  "mlock": true,
  "n_batch": 1024,
  "n_ubatch": 512
}
```

**目标**：支持长上下文，Flash Attention 降低内存。

#### 场景 D：成本优先（小内存节点）

```json
{
  "ctx_size": 2048,
  "threads": 2,
  "flash_attn": true
}
```

**目标**：最小内存占用，单机部署小模型。

### 5.2 调度策略选择

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| 多副本高可用 | `spread`（默认） | 副本分散到不同节点，单节点故障不影响全部副本 |
| 腾出大内存节点 | `binpack` | 打包到同节点，留出大内存节点供大模型部署 |
| 流水线并行 | 自动（网络感知） | `select_pipeline_nodes` 自动选高带宽节点 |
| RPC 内存池化 | 自动（指令集优先） | `select_rpc_master` 自动选强指令集 Master |

### 5.3 性能问题排查

| 现象 | 可能原因 | 排查建议 |
|------|---------|---------|
| 吞吐低于预期 | `cont_batching` 未开启 | 检查 `backend_parameters` 配置 |
| TTFT 过高 | 无前缀缓存复用 | 配置 `cache_reuse=256` |
| 延迟抖动大 | 内存 swap | 开启 `mlock=true`，检查可用内存 |
| 流水线无加速 | 网络带宽不足 | 检查 Worker 心跳上报的 `network_bandwidth`，需 ≥10Gbps |
| RPC 无加速 | 误用 RPC 期望加速 | RPC 仅解决内存不足，不加速（规划文档 §9 难点1） |
| AVX-512 节点未被优先 | 指令集未正确上报 | 检查 Worker 心跳的 `instruction_sets` 字段 |

## 6. 向后兼容性保证

阶段5 所有优化参数**全部可选**，未配置时保持阶段4 行为：

| 模块 | 阶段4 行为 | 阶段5 默认（未配置参数） |
|------|-----------|----------------------|
| `build_common_args` | — | 仅生成 `--ctx-size 4096 --threads <allocated_cpu_cores or 4>` |
| `score_placement` | 无策略 | 默认 `spread`，无指令集/网络感知（单候选直接返回） |
| `select_rpc_master` | 按内存降序 | 单候选直接返回，多候选拼接集+内存打分 |
| `select_pipeline_nodes` | 按 CPU 核心降序 | 指令集相同时退化为 CPU 核心打分 |

**升级建议**：现有模型无需修改 `backend_parameters`，可在适当时机逐步启用优化参数。
